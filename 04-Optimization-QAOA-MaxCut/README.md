# 04-Optimization-QAOA-MaxCut

QAOA and RQAOA benchmark package for the Max-Cut problem.

This project focuses on a clean, reproducible workflow for:

- generating graph instances
- building Max-Cut cost operators
- optimizing QAOA parameters against backend-reported expected objectives
- running reduction-based RQAOA
- comparing against exact classical solutions
- generating benchmark plots and CSV summaries

## What Changed

This revision fixes the major correctness issues in the earlier version:

- The variational objective now minimizes `-(offset + <interaction term>)`.
- The direct graph-based circuit builders now use the correct weighted
  `RZZ(-gamma * w_ij)` angles.
- The QUBO-derived Hamiltonian path is mathematically identical to the main
  Ising mapping instead of introducing spurious linear `Z` terms.
- Runtime execution reports both the interaction expectation and the recovered
  Max-Cut objective.
- QAOA now reports the expected cut value as its primary output, with a
  representative sampled bitstring tracked separately.
- Representative sampled outputs now use the most likely observed bitstring;
  the best sampled cut is tracked separately so reporting does not silently
  flatter the optimizer.
- The benchmark and integration paths now run through the execution layer
  instead of bypassing it with ad hoc statevector calls.
- Depth sweeps can warm-start `p+1` from the previous depth's learned
  parameters instead of restarting every layer from scratch.
- Objective averaging, noisy-candidate re-evaluation, and plateau diagnostics
  are configurable from `config/experiment_config.yaml`.
- The noisy-simulator path now falls back to a matching fake IBM backend when
  live Runtime credentials are unavailable, so NISQ proxy runs remain local
  and reproducible.
- Hardware-feasibility analysis now reports transpiled depth, entangling-gate
  growth, and shot-budget pressure in `results/hardware_feasibility.csv`.
- The repo now includes a held-out experimental study across multiple graph
  families and seeds, with explicit tuning/evaluation splits, classical
  heuristic baselines, confidence intervals, and paired significance tests.
- The held-out study now also includes stronger and fairer comparisons:
  Goemans-Williamson SDP rounding plus budget-matched classical baselines that
  use the same objective-evaluation count as tuned QAOA.
- The study now also exports pairwise win/loss summaries and Holm-corrected
  p-values so weak raw effects are not overstated.
- The default noisy benchmark now has a repeated-seed robustness pass instead
  of relying on a single lucky optimization run.
- The repo now produces an explicit scientific verdict artifact summarizing
  whether the results are strong, weak, misleading, or invalid.
- The repo now also generates a publication-positioning artifact that frames
  the project honestly as a benchmark/negative-results software artifact
  instead of overstating algorithmic novelty.
- RQAOA can estimate pair correlations from sampled counts instead of relying
  only on exact statevectors.
- RQAOA tracks the constant terms generated by opposite-spin eliminations, so
  reduced instances remain mathematically equivalent to the constrained
  original objective.
- The default config now uses a weighted communication-mesh instance.
- Notebook derivations and examples have been updated to match the corrected
  formulas.
- Real benchmark artifacts are generated and checked into `results/`.

## Research Scope

The repo is best viewed as a benchmark-oriented research toolkit and a
negative-results artifact, not as a novel algorithmic contribution. It is
useful for studying and validating the QAOA/RQAOA pipeline on small and medium
synthetic Max-Cut instances.

The current "robot communication mesh" framing is now implemented as a weighted
communication-graph proxy with domain-style edge attributes. It is still not a
fully domain-faithful robotics optimization model.

## Max-Cut Objective

For an undirected graph `G = (V, E)` with optional edge weights `w_ij`, the
Max-Cut objective is

`C = sum_(i,j in E) w_ij * (1 - Z_i Z_j) / 2`

In this codebase, `HamiltonianBuilder.build_maxcut_hamiltonian(...)` returns:

- the diagonal ZZ interaction term as a `SparsePauliOp`
- the constant offset separately

To recover the cut objective for a quantum state, use:

`cut_value = offset + <interaction_term>`

Because the optimizers minimize by default, the project minimizes
`-(offset + <interaction_term>)`.

The primary QAOA result is the expected cut value of the optimized quantum
state. A single bitstring is reported only as a representative sampled output,
not as the quantity that was optimized.

## Reproducibility Model

Artifact generation is now run-scoped instead of overwriting the project state
without traceability.

- `generate_artifacts.py` is a thin entrypoint over `src/artifact_pipeline.py`.
- `src/artifact_manager.py` writes each run into `results/runs/<run_id>/` and
  mirrors the latest stable copies into `results/`.
- `src/provenance.py` records a `results/run_manifest.json` file containing the
  run id, config hash, git commit, Python version, platform, and package
  versions used for the artifacts.
- `src/artifact_schema.py` defines typed artifact rows so CSV/JSON outputs have
  an explicit schema instead of ad hoc dict payloads.

## Current Configured Benchmark

The current config uses:

- graph family: weighted `communication_mesh`
- number of nodes: 6
- optimizer: SPSA
- QAOA depths: `p = 1`
- quantum execution mode: `noisy_simulator`
- target backend proxy: `ibm_brisbane`
- primary objective: expected cut value
- optional risk-sensitive objective: CVaR with configurable `cvar_alpha`
- warm starts across depths: enabled
- representative bitstring analysis: `same_backend`
- RQAOA: disabled by default in the NISQ proxy benchmark

Current generated metrics in `results/metrics.csv`:

| Method | Depth | Expected Cut | Representative Sampled Cut | Best Sampled Cut | Approx. Ratio | Runtime (s) |
|---|---:|---:|---:|---:|---:|---:|
| Exact | 0 | 4.4724 | 4.4724 | 4.4724 | 1.0000 | 0.0005 |
| QAOA | 1 | 2.9925 | 2.7901 | 4.4724 | 0.6691 | 24.8027 |

These are noisy-simulator results on a fake-`ibm_brisbane` hardware proxy.
The expected-cut column is the algorithmically relevant quantity for QAOA.
The representative sampled-cut column is the most likely measured bitstring,
not the best bitstring seen in the sample set. The best-sampled column is
reported separately because it can look better than the optimized expectation.

The communication-mesh generator now assigns domain-style edge attributes such
as distance, latency, reliability, interference, and bandwidth, then maps the
resulting weighted graph into the Max-Cut Hamiltonian. The mathematical details
are written out in `docs/mathematical_formulation.md`.

The repeated-seed robustness summary in `results/benchmark_robustness_summary.csv`
is the more honest benchmark view:

| Depth | Mean Ratio | 95% CI | Mean Sample Gap | Exact Sample Hit Rate | Mean Runtime (s) |
|---|---:|---:|---:|---:|---:|
| 1 | 0.6856 | [0.6526, 0.7064] | -0.2278 | 0.0000 | 25.2558 |

That robustness pass makes the core benchmark interpretation much clearer:
the weighted noisy QAOA state is stable enough to benchmark, but it is still
well below exact performance and its representative measured bitstrings are not
reliably better than the optimized expectation.

The hardware-feasibility report in `results/hardware_feasibility.csv` shows
that this default proxy benchmark stays at 6 qubits, logical depth 11,
transpiled depth 126, and 43 transpiled two-qubit gates with an estimated
13,312-shot budget. That is still small and noisy, but it is far closer to
realistic NISQ conditions than the earlier exact-statevector benchmark.

The latest run metadata for these artifacts is captured in
`results/run_manifest.json`, and the complete run-scoped copies are preserved
under `results/runs/<run_id>/`.

## Held-Out Study

The repo now also runs a separate scientific study using:

- three graph families: `communication_mesh_8_3`, `d_regular_8_3`, and `erdos_renyi_8_0.5`
- held-out tuning seeds: `101, 202, 303`
- held-out evaluation seeds: `404, 505, 606, 707, 808, 909, 1001`
- candidate QAOA hyperparameters: depth `p in {1, 2}` and restart count `in {1, 2}`
- objective mode: expected value by default, with CVaR available through config
- baselines: exact, greedy, local search, repeated random cut, Goemans-Williamson SDP rounding, and budget-matched random/hill-climb heuristics

The tuning sweep in `results/study_candidate_search.csv` selects:

- QAOA depth: `p = 2`
- random restarts: `2`
- max iterations: `40`

Held-out evaluation summary from `results/study_method_summary.csv`:

| Family | QAOA tuned | Goemans-Williamson | Hill climb, budget matched |
|---|---:|---:|---:|
| Communication mesh | 0.7828 | 1.0000 | 0.9962 |
| D-regular | 0.8632 | 1.0000 | 1.0000 |
| Erdos-Renyi | 0.8341 | 1.0000 | 1.0000 |

Paired significance tests in `results/study_significance.csv` are blunt:

- Raw permutation p-values still trend against QAOA on all three families.
- After Holm correction, the study still does **not** support a statistically
  significant QAOA advantage claim.
- The pairwise win/loss table in `results/study_pairwise_summary.csv` is still
  practically unfavorable to QAOA: on the new communication-mesh family it
  loses `7/7` held-out instances against every stronger baseline.

The budget-focused summary in `results/study_budget_summary.csv` is especially
damning: with the same mean objective-evaluation count (`40`) as tuned QAOA,
the budget-matched hill climber reaches mean approximation ratios of `0.9962`,
`1.0000`, and `1.0000` across the three families, while QAOA remains at
`0.7828`, `0.8632`, and `0.8341`.

The explicit review artifact in `results/results_verdict.json` therefore labels
the current evidence as `weak` with `medium` misleading-risk. The companion plots
`results/study_significance_heatmap.png`, `results/study_budget_fairness.png`,
and `results/sample_gap_analysis.png` make those failure modes visible at a
glance.

The publication-positioning artifact in `results/publication_positioning.md`
classifies the repo as a `benchmark_negative_results_artifact` and recommends
framing it as a workshop-level benchmark or negative-results paper rather than
as a novel algorithm paper.

## Main Modules

- `src/graph_generator.py`
  Builds D-regular, Erdos-Renyi, Barabasi-Albert, and weighted communication-mesh graphs.
- `src/hamiltonian_builder.py`
  Converts graphs into Max-Cut interaction operators and offsets.
- `src/qaoa_circuit.py`
  Builds single-layer and multi-layer QAOA circuits.
- `src/qaoa_optimizer.py`
  Tracks objective evaluations, exposes a backend-aware Max-Cut QAOA workflow,
  supports expected-value or CVaR objectives, warm starts, reevaluation
  controls, and separates expected values from representative versus optimistic
  sampled artifacts.
- `src/rqaoa_engine.py`
  Performs reduction-based RQAOA with weighted graph updates and exact fallback.
- `src/runtime_executor.py`
  Supports exact local evaluation, noisy Aer sampling, and Runtime execution,
  with separate interaction and objective reporting, sampled probability
  distributions, and fake-backend proxy fallback for IBM hardware studies.
- `src/hardware_analysis.py`
  Estimates transpiled depth, entangling-gate growth, and shot budgets on a
  target backend.
- `src/experimental_study.py`
  Runs held-out tuning and evaluation across graph families, including the
  weighted communication-mesh family, aggregates results, computes paired
  significance tests against classical baselines, exports pairwise win/loss
  summaries, and produces a budget-matched study slice for fair black-box
  comparison.
- `src/artifact_pipeline.py`
  Orchestrates benchmark execution, study generation, artifact writing, and
  provenance capture behind the thin `generate_artifacts.py` entrypoint.
- `src/artifact_manager.py`
  Writes run-scoped outputs and mirrors stable latest artifacts into
  `results/`.
- `src/artifact_schema.py`
  Defines typed records for benchmark metrics, study outputs, robustness
  summaries, and hardware-feasibility tables.
- `src/evaluation_metrics.py`
  Computes approximation ratios, benchmark summaries, confidence intervals, and
  multiple-testing corrections.
- `src/provenance.py`
  Captures config hashes, git state, interpreter metadata, and package
  versions for each generated run manifest.
- `src/results_review.py`
  Runs repeated-seed benchmark robustness analysis and generates an explicit
  scientific verdict from the benchmark and held-out study artifacts.
- `src/visualization.py`
  Saves plots for landscapes, convergence, cuts, significance heatmaps, budget
  fairness, and sampled-vs-expected gap analysis.

## Limitations

- The notebooks are explanatory and are not treated as the source of truth for
  regression testing.
- The small hardware proxy benchmark is still a single configured graph.
- The held-out study is stronger, but it still uses only three small 8-node
  graph families.
- The repeated-seed robustness analysis currently covers only the configured
  benchmark graph, not the full multi-family study.
- The default benchmark is now a hardware proxy, but it still uses a fake
  backend rather than live queue-calibrated IBM hardware.
- The project now has a stronger empirical framing, but the generated
  `publication_positioning.md` still correctly classifies it as benchmark
  software plus negative results, not algorithmic novelty.
- There is still no project-owned analytic gradient pipeline; the default loop
  remains derivative-free unless you opt into optimizer-library methods.
- The RQAOA implementation is a pragmatic benchmark version, not a claim of a
  novel reduction method.
- The communication-mesh objective is more realistic than the earlier
  D-regular surrogate, but it is still a graph-weight proxy rather than a full
  robotics networking model.

## Next Research Upgrades

- Extend the repeated-seed robustness analysis to the multi-family study.
- Add stronger classical baselines for medium-sized weighted graphs and include
  them in the held-out study, not just the exact small-instance regime.
- Add live IBM Runtime experiments with shot-accounted Sampler outputs.
- Add a project-owned analytic or parameter-shift gradient path so optimization
